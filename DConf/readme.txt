# DConf

Логически код можно разделить на следующие модули:

* **data_model** — классы конфигурации шаблонов устройств, структуры проекта, свойства sybus параметров устройств.
* **db** — классы для работы с sqlite3.
* **device_operations** — классы, в которых реализована логика взаимодействия с устройством.
* **file_managers** — классы, реализующие понятие менеджер файлов конкретного функционала. 
* **gui** — классы, описывающие весь пользовательский интерфейс приложения. Структура настроек устройств, меню, формы, элементы управления, диалоговые окна.
* **journal** — классы журнала и записей для реализации механизма логирования сообщений.
* **report** — классы для генерации отчёта конфигурации устройства
* **updater** — класс для проверки наличия новой версии и запуска обновления самого приложения.
* **utils** — Утилиты. вспомогательные классы.
* ** DConf ** - Основной оконный класс приложения
* ** dc_proj_model ** - Мультифункциональный класс!!!

---

## data_model

Файлы модуля находятся в папке:

```
/data_model/
```

**Описание:**

1. В этой папке содержатся классы: конфигурации шаблонов устройств, структуры проекта, работы с параметрами устройств. все накидано в одну кучу
2. Все сохраняемые классы конфигурации устройства, реализуют паттерн активная запись.

**Зависимости:**

* Зависит от модуля db. Сами классы реализуют вызов транзакций для внесения изменений в таблицы базы данных sqlite, через классы из db.

## device_operations

Файлы модуля находятся в папке:

```
/device_operations/
```

**Описание:**

* Классы в которых реализована логика взаимодействия с устройством, как отдельные операции(запись/чтение конфигурации, перезагрузка, сброс, форматирование, обновление и т.д.)
* Есть базовый класс операции (DcIDeviceOperation), в котором прописан базовый алгоритм взаимодействия с устройством:
- Предварительные действия
- Установка соединения
- Выполнение взаимодействия с устройством (чтение/запись параметров)
- Разрыв соединения
- Пост обработка.
Все конкретные операции наследуются от данного класса и переопределяют необходимые методы, а также используют методы для уведомления о состоянии операции, ее статус, ошибки, предупреждения, информация и т.д.

**Зависимости:**

* data_model - для работы с конфигурацией устройства, получения/установки значений параметров. 
* db - для загрузки вычитанного с устройства файла конфигурации.
* библиотека dpc - для взаимодействия с устройством.

## file_managers

Файлы модуля находятся в папке:

```
/file_managers/
```

**Описание:**

Для конфигурирования некоторых функций устройства, используются конфигурационные файлы. Для каждого функционала файлы имеют разную структуру, 
количество и записываются в разные конфигурационные параметры. Для удобства отслеживания путей к этим файлам, введено
понятие файловый менеджер к функционалу. Они нужны, чтобы автоматически по конфигурации определять какие файлы необходимо записать в устройство при записи
или прочитать из устройства при чтении. Базовый класс файлового менеджера(IFileManager) реализует общие методы. Для каждого функционала где
используется дополнительные конфигурационные файлы, пишется свой менеджер.

**Зависимости:**

* data_model - для работы с конфигурацией устройства, получения/установка значений параметров.

## gui

Файлы модуля находятся в папке:

```
/gui/
```

**Описание:**

классы описывающие весь пользовательский интерфейс приложения. Структура настроек устройств, меню, формы, элементы управления, диалоговые окна.


### Подмодули

1. ** /gui/dialogs ** — Диалоговые окна, используемые в приложении.
2. ** /gui/editors ** — Классы для автоматического создания элементов управления по типу параметра, а также их расположению на виджетах/формах.
- *Editor - набор классов реализующий элемент управления параметром. По заданию пользователя или в зависимости от типа параметра может быть разным типом (текстовое поле, спин бокс, чекбокс, список).
- В зависимости от типа параметра автоматически устанавливает диапазон значений, маску ввода, подкрашивается в "не валидный цвет" при несоответствующем значении списку элементов.
- EditorsManager - класс, форматирующий расположение и отображение элементов управления на форме.  
3. ** /gui/forms ** — Формы настроек устройства:
- DcMenu.h - Иерархическое меню настроек устройства. Через макрос определен порядок подпунктов меню, и их иерархия.
- DcFrom.h - Базовый класс формы. Все формы, наследуются от данного класса. Каждая форма через макрос регистрируется в фабрике и прикрепляется к конкретному подпункту меню настроек устройства.
- DcFromFactory.h - Фабрика форм. 
- Если для какого то подпункта меню в фабрике не зарегистрирована форма, то генерируется дефолтная(пустая или с логотипом) форма.

**Зависимости:**

* data_model - для работы с конфигурацией устройства, получения/установка значений параметров. 
* file_managers - для работы с файлами
* report - для создания отчета об конфигурации устройства согласно дереву настроек устройства.

## dc_proj_model

Файлы:

```
/dc_proj_model.h
```

**Описание:**

- Класс отвечает за отображение структуры дерева проекта в панели дерева проекта.
- Создание/Удалении папок в проекте
- Добавление/Удаление/Копирование/Импортирование/Экспортирование конфигураций устройств.
- Загрузка конфигурации устройств из файлов. 
- Обновление конфигурации устройства из соответствующего шаблона конфигурации. 

**Зависимости:**

* data_model - загрузка структуры проекта (папки, конфигурации устройств) и создание дерева в панели отображения дерева проекта. Обновление конфигураций по шаблону конфигурации.
* db - загрузка конфигураций устройств из файлов проекта.
